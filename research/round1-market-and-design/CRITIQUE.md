## GAPS
- Localization is unsolved: .uetkx string literals never enter the FText gather pipeline (no LOCTEXT/NSLOCTEXT namespaces, no string-table story), interpreted mode can only produce culture-invariant text from raw strings, the API sketch itself uses INVTEXT, and there is no live-culture-change trigger to re-render mounted trees. As written, a shipping game cannot localize UI built with this framework.
- No brush/asset-reference story: FSlateBrush needs textures/materials, but nothing says how V::Image or the style dicts reference assets (soft object paths? sync vs async load?), nor how UTexture/UMaterial objects referenced by brushes held in non-UObject fiber code are kept alive against GC (FGCObject/AddReferencedObjects is never mentioned outside the two named TStrongObjectPtr cases). This is the classic Slate-in-C++ crash/leak class.
- Animation is entirely absent: no FCurveSequence analogue, no UMG-animation interop, no mount/unmount transitions or interpolated props. Noesis, Gameface, and UMG all have motion tooling; a game-UI framework without any animation primitive is a v1 adoption blocker, and retrofitting it touches the reconciler (exit animations require delayed unmount).
- No list virtualization design: keyed children are materialized 1:1 as widgets, list views are excluded from pooling and their columns are construct-only args, so a 5,000-row inventory creates 5,000 SWidgets — exactly the 1,000-children/hitching problem the demand research cites. Mapping declarative children onto SListView's OnGenerateRow/recycling model (the Godot `items` seam) is the hard, missing adapter.
- No portal mechanism: tooltips, context menus, combo-box dropdowns, drag previews, and modal SWindows live outside the widget tree via FSlateApplication::PushMenu/AddWindow; the designs define no React-portal analogue, no ownership or cleanup rules for out-of-tree content spawned by a component that then unmounts.
- Focus lifecycle across diffs is undefined: reconstruct-on-prop-mask replacement or a keyed reorder can destroy or move the currently focused widget mid-commit. Nothing specifies capture/restore of focus, per-user focus (FSlateUser, split-screen), or how Slate's navigation config behaves when siblings are permuted across slots.
- Commit-phase reentrancy has no contract: Slate setters fire delegates synchronously (SetText → OnTextChanged, SetContent → focus-change events), so a user handler can call a hook setter during the commit phase — which is neither the mid-render-restart case (25-guard) nor the normal pre-tick case. The editable-text skip-when-equal note covers one instance of a general problem.
- No threading contract: FieldNotify delegates are documented game-thread-only, but nothing states whether hook setters/RequestUpdate may be called from async-load or network callbacks, and no AsyncTask(ENamedThreads::GameThread) marshaling or checkf(IsInGameThread()) policy is defined.
- Per-render allocation budget unexamined: every render heap-allocates one TUniquePtr<FRUIPropsBase> per vnode, plus TArray<FRUINode> children, plus TFunction captures — thousands of allocations per frame on large trees at 60fps. The slab allocator covers fibers only; no vnode frame-arena, no stat group, no ported bench suite, no target numbers, and no measurement plan comparing against UMG/MVVM baselines despite perf being a core pitch.
- World-space UI is unaddressed: UWidgetComponent / render-target UI and world-space hit testing (a standard game-UI requirement) appear nowhere; presumably UReactiveWidget-in-a-WidgetComponent works, but retainer-box/render-target invalidation interactions are unstated.
- UMG Designer design-time behavior: UReactiveWidget::RebuildWidget executes in the Designer preview, meaning components with hooks, passive effects, and VM subscriptions run at design time in an editor world — needs an explicit design-time stub/preview mode; not mentioned.
- Blueprint-only authoring gap: markup handlers require a C++ UFUNCTION logic class or a C++ registration macro; whether @logic can be a Blueprint class (FindFunction does resolve BP functions) is never stated. Fab's largest buyer demographic is Blueprint-heavy, and the only BP-free logic is the DSL's set() — the addressable market vs the authoring requirement is unexamined.
- Markup source-of-truth contradiction: Design 3 says the spec 'lives inside URuiMarkupAsset : UPrimaryDataAsset (retains source text)' — a binary .uasset, which kills the 'text files you can diff' pitch — while the watcher watches loose *.uetkx files. The import pipeline (UFactory/reimport, cook rules, redirects), and whether cooked builds ship the spec, the generated C++, or both, is unspecified. The two documents also disagree on the extension itself (.uitkx in Design 2, .uetkx in Design 3), evidence the designs were never reconciled.
- Codegen file ownership: where generated .g.h/.g.cpp files live (user game module? a generated module?), who wires Build.cs and include paths in the buyer's project, how stale generated files are cleaned, and how the markup compiler binary itself is built/distributed for Windows/Mac/Linux build hosts and CI is all unspecified.
- Input consumption policy: defaulting void callbacks to FReply::Handled() silently swallows events (breaking bubbling, drag detection, touch pass-through); there is no drag-and-drop API, no keyboard-shortcut story, and no policy for analog/gamepad events outside CommonUI delegation.
- Interpreted-mode runtime errors: last-good-spec handles parse errors, but a DSL evaluation error mid-render (bad property path, items[selected] out of range after a data change) has no defined behavior, and the structural error boundaries explicitly cannot catch a child render crash.
- Lifetime vs world teardown: URuiSubsystem's type (GameInstance vs World subsystem) is never chosen; roots and the widget pool hold TSharedPtr<SWidget> and TStrongObjectPtr<UUserWidget> across level travel and PIE end — the shutdown-ordering crash class (ReleaseSlateResources vs subsystem deinit vs GC) is unaddressed.
- Accessibility: Slate's screen-reader/accessibility layer (FSlateAccessibleWidget, platform cert requirements) is never mentioned, despite 'accessibility already understands our widgets' appearing in the positioning copy.
- Styling is deferred but load-bearing: every markup example uses style dicts ({"bg_color", "font_size"}) that Design 1 explicitly defers to post-v1; who converts CSS-ish dicts into construct-heavy FSlateWidgetStyle structs is unanswered, and since style bits sit on the reconstruct-on-prop mask, any style tweak means widget replacement — focus/state loss plus the expensive ChildOrder invalidation, undermining the hot-reload demo where style tweaking is the primary use.
- Team-size vs scope math is absent: four modules, a two-backend markup toolchain (compiler + interpreter + expression DSL), an LSP, an inspector, bidirectional MVVM bridges, CommonUI citizenship, and a 3-engine-version CI matrix is multiple engineer-years, with no v1 cut-line or resourcing plan — against research showing plugin support is 'a full time job', mean Fab publisher earnings ~$1.2K/yr, and a UE5→UE6 transition (5.8 is the last UE5 release) landing mid-way through the plugin's obligatory support window.
- Hot-reload state-migration rule missing: adding, removing, or reordering a `state` declaration in markup shifts positional hook slots, so 'state survives' silently becomes state scrambled or type-mismatched; no slot-migration, signature-hash reset, or even documented reset-on-state-edit rule is defined.

## CHALLENGED CLAIMS
### FieldNotify (INotifyFieldValueChanged) is consumable with zero dependency on the ModelViewViewModel plugin, making any Epic-MVVM/MDViewModel VM or stock widget a data source.
VERDICT: CONFIRMED
Research fact places INotifyFieldValueChanged, FFieldId, delegates, and UHT-generated descriptors in the engine runtime FieldNotification module; web check confirms UFieldNotificationLibrary (used for SetPropertyValueAndBroadcast in the UMG-interop path) lives in the Engine module, not the plugin (https://dev.epicgames.com/documentation/en-us/unreal-engine/API/Runtime/Engine/UFieldNotificationLibrary, https://dev.epicgames.com/documentation/en-us/unreal-engine/API/Runtime/FieldNotification). One caveat the design handles correctly: authoring macros (UE_MVVM_SET_PROPERTY_VALUE) and UMVVMViewModelBase/global collection are plugin-side, hence the gated bridge module. The design consumes, it doesn't author — the claim holds.

### Commit-time reorder via DetachWidget/AttachWidget permutation across existing slots 'minimizes ChildOrder invalidations (the expensive reason)'.
VERDICT: DOUBTFUL
The mechanics exist (UE 5.0 slot refactor added FSlotBase::AttachWidget/DetachWidget per the research), but attaching/detaching a widget re-parents it, which marks the parent ChildOrder-dirty exactly as remove+insert would — both paths end with the fast-path flattened widget list rebuilt, which the research rates as the most expensive invalidation and links to live UE 5.7 crash reports in FSlateInvalidationRoot::ClearAllFastPathData under churn. What permutation actually saves is slot-object construction/destruction, not the ChildOrder invalidation; the perf rationale as stated is oversold and unmeasured. Doing reorders only on structural frames is right, but the headline benefit is misattributed.

### Diff-driven constant-value setter writes (never delegate TAttributes) keep widgets non-volatile and fast-path-valid — 'the ideal Slate citizen'.
VERDICT: CONFIRMED
This matches Epic's documented guidance from three independent research facts: legacy polled TAttribute delegates are 'high memory overhead and not cache friendly', UE 5.0 TSlateAttribute exists precisely to make attributes invalidation-registered, engine setters self-invalidate, and Epic's stated at-scale guidance is event-driven setter writes. Global Invalidation (VALORANT's 15% FPS win) rewards exactly this pattern. The one unproven corner is whether per-frame prop diffing itself stays cheap at scale (see the missing perf-measurement plan), but the Slate-citizenship claim is correct.

### Runtime-interpreted markup delivers a <1 s, state-preserving, mid-PIE reload loop — 'state survives because hook state lives in fibers keyed by position/key'.
VERDICT: PLAUSIBLE
The mechanics are sound (IDirectoryWatcher, spec swap, one reconciler downstream, last-good-spec on parse error), and Noesis proves state-preserving markup reload commercially. But the guarantee is overstated: Noesis preserves state because state lives in the ViewModel, whereas here state lives in positional hook slots — any edit that adds/removes/reorders a `state` declaration or a conditional branch shifts slot indices and scrambles or type-mismatches state. 'State survives' is true only for structure/style/expression edits that don't touch the hook signature, and no document defines the behavior when they do. The <1 s claim also assumes style dicts work, which Design 1 defers.

### PreBuildSteps-driven markup→C++ codegen is a proven, low-risk ship path (unreal.hx/protoc precedent), with 'shape-stable' headers keeping Live Coding on its safe path.
VERDICT: DOUBTFUL
UBT's cached makefiles gather the source-file list before pre-build steps execute; adding or removing source files invalidates the makefile and requires re-gathering/project regeneration (https://dev.epicgames.com/documentation/unreal-engine/build-configuration-for-unreal-engine, https://forums.unrealengine.com/t/pre-post-build-step-in-ubt/355586) — so 'create a new component' generates a .g.cpp the same build won't compile. The precedents regenerate known file sets, which is the easy case. The design's own research notes header churn is Live Coding's documented worst case with an active UE 5.8 recompile-loop regression, and 'regenerated only on prop-signature change' is exactly the change users make constantly while iterating on a component's props. Add unaddressed distribution of the compiler binary per build-host OS, and this is a fragile path presented as solved.

### Coalescing re-renders to one per frame 'natively gives MVVM's Delayed execution semantics — the mode that benchmarks 76x faster than Immediate' (headline interop claim).
VERDICT: DOUBTFUL
The 976.7 ms vs 12.8 ms community benchmark measures batching of MVVM's compiled bindings — direct property-write execution. ReactiveUI's coalesced pass re-runs component functions, allocates a vtree, and diffs it per dirty fiber: strictly more work per update than a compiled binding write. Borrowing the 76x figure implies a ReactiveUI-vs-Immediate comparison that was never run; the defensible claim is only 'we batch to one pass per frame like Delayed mode'. Whether a re-render+diff beats even Immediate-mode bindings at realistic update rates is exactly what the missing benchmark plan should establish before this becomes marketing copy.

### The Fab distribution model works for this: MIT core on GitHub plus a $29.99–$59.99 one-time paid editor-tooling module, Epic-built binaries, three-engine-version support.
VERDICT: DOUBTFUL
Mechanically compliant (Fab compiles submitted source, 88/12 split, price anchors exist, VRExpansion/Voxel precedent) but economically and temporally oversold. Research: no subscriptions on Fab so all revenue is one-time against a permanent 'full time job' support burden; forum consensus doubts even EUR 400-500/month for code sellers; mean publisher earnings ~$1.2K/yr; code-plugin review latency of weeks-to-months directly conflicts with the must-support-latest-version rule at Epic's ~2 releases/year. Strategically worse: UE 5.8 is the last UE5 release, so the mandatory ~18-month support window runs straight into the UE6 transition (EA end-2027), where the designs assume without evidence that Slate carries over unchanged — while Epic gives away SlateIM, ViewModel, and CommonUI, compressing the paid layer to exactly the editor tooling a free VS Code extension (this family's own precedent) undercuts.

### Passing FRUIContext& into every component makes calling a hook outside render 'a compile error, not a runtime assert'.
VERDICT: DOUBTFUL
It only makes hooks uncallable without a context reference in scope; nothing stops the ubiquitous failure mode — capturing Ctx (or a struct containing it) in an OnClicked lambda or stored callback and invoking hooks after render, which compiles cleanly and corrupts the slot cursor at runtime. The design tacitly concedes this by keeping a 'debug-only thread-local to assert in-render usage'. The honest claim is 'narrows the error class and removes per-hook TLS lookups'; 'compile error' is an overclaim that will mislead adopters about a real footgun, especially since the same document's event system encourages exactly such lambda captures.
