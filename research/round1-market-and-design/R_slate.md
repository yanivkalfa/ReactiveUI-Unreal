## Slate as a host for a React-style reconciler — research findings

### 1. Creation/mutation API
`SNew(WidgetType)` expands to `MakeTDecl<WidgetType>(#WidgetType, __FILE__, __LINE__, RequiredArgs::MakeRequiredArgs(__VA_ARGS__)) <<= WidgetType::FArguments()` — it heap-allocates the widget as a `TSharedRef`, builds a stack `FArguments` struct, then calls `Construct(const FArguments&)`. `SAssignNew(Var, Type)` is identical plus `.Expose(ExposeAs)` to capture the pointer. The file/line capture feeds the Widget Reflector. Args are declared between `SLATE_BEGIN_ARGS`/`SLATE_END_ARGS` (FArguments inherits `TSlateBaseNamedArgs<WidgetType>`, members are `_Name`): `SLATE_ARGUMENT` (plain value, construct-time only), `SLATE_ATTRIBUTE` (`TAttribute<T>`, constant or delegate-bound), `SLATE_EVENT` (delegate), `SLATE_SLOT_ARGUMENT`/`SLATE_NAMED_SLOT` (children).

**Post-construction mutation is fully supported and is exactly what UMG itself does**: every `UWidget` holds a `TSharedPtr` to its underlying SWidget ("Cached pointer to the underlying slate button owned by this UWidget"), constructs it once in `RebuildWidget()`, and thereafter pushes changes via setters (`SynchronizeProperties`). So a diff-based host is architecturally precedented inside the engine. Concrete setters: base `SWidget` — `SetVisibility`, `SetEnabled`, `SetRenderTransform`, `SetRenderOpacity`, `SetToolTipText`, `SetCursor`, `ForceVolatile`, `Invalidate(EInvalidateWidgetReason)`, `MarkPrepassAsDirty` (UE 5.1+; `InvalidatePrepass` deprecated), `SlatePrepass`. Leaves: `STextBlock::SetText/SetFont/SetColorAndOpacity/SetJustification`, `SImage::SetImage`. Content wrappers: `SBorder::SetContent`, `SBox::SetContent/SetWidthOverride`, `SButton::SetContent/SetButtonStyle/SetContentPadding/SetOnClicked`. `SCompoundWidget`'s `ChildSlot` can be re-targeted (`ChildSlot.AttachWidget(...)` / `[ ... ]`). Caveat: not every FArguments arg has a setter (style structs on complex widgets, `SListView` column setups are effectively construct-only); a host needs a per-widget "reconstruct on this prop" table — same pattern as your `RUIHost` seam. One known wart: `STextBlock::SetTextStyle` historically didn't fully re-apply (hyaniner.com bug writeup, 2024-05).

### 2. Dynamic children
Panels store children in `TPanelChildren<SlotType>` (array of slot objects; each slot owns its child `TSharedRef` plus layout params). Public APIs: `SVerticalBox/SHorizontalBox` (both are `SBoxPanel`) — `AddSlot()`, `InsertSlot(int32 Index)`, `RemoveSlot(const TSharedRef<SWidget>&)`, `ClearChildren()`, `NumSlots()`, `GetSlot(i)`. UE 5.0 refactored slot construction: `AddSlot` returns `FScopedWidgetSlotArguments`, and slot params have runtime setters (`SetPadding`, alignment, size rules); `FSlotBase::AttachWidget(TSharedRef<SWidget>)` / `DetachWidget()` swap a slot's child in place (the declarative `[ ]` is just `operator[]` → `AttachWidget`). `SOverlay` — `AddSlot(ZOrder)`, `RemoveSlot`, slots draw in declaration order unless per-slot `ZOrder` differs ("larger ZOrder draws above; equal ZOrder draws in add order"); community code shows a `ChangeSlot` pattern that mutates ZOrder and re-sorts. `SGridPanel::AddSlot(Column, Row, Layer)`; `SCanvas`/`SConstraintCanvas` have `AddSlot`/`ClearChildren` with position/size slot attributes. **Reordering: there is no public MoveSlot on the box panels** — a reconciler must remove+insert (or detach/attach across slots, which is cheaper than destroying widgets). Cost model: any child add/remove fires `ChildOrder` invalidation, the most expensive reason ("High" per Epic docs; "can cause significant performance bottlenecks if performed every frame") because it rebuilds the child list and re-lays-out the whole subtree — and under global invalidation forces the invalidation root to rebuild its flattened fast-path widget list (`FSlateInvalidationRoot::BuildFastPathWidgetList`). Coconut Lizard's Slate 101 leaderboard confirms holding a `TSharedRef` to a panel and calling `AddSlot()` at runtime is the normal dynamic pattern.

### 3. Invalidation model
`EInvalidateWidgetReason` bit flags: `Layout` (1<<0, desired size changed — triggers prepass, costliest routine reason), `Paint` (1<<1, visual only, geometry cache stays valid), `Volatility` (1<<2), `ChildOrder` (1<<3, implies Layout+Prepass), `RenderTransform` (1<<4), `Visibility` (1<<5, implies Layout), `Prepass` (1<<7, re-measure recursively); combos `PaintAndVolatility`, `LayoutAndVolatility`. `SWidget::Invalidate(reason)` routes through `FastPathProxyHandle.MarkWidgetDirty()` to the owning `FSlateInvalidationRoot`, which keeps a dirty list processed next frame. **What's automatic**: engine widget setters already invalidate — UE5 `TSlateAttribute` members are registered with a reason (`SLATE_ADD_MEMBER_ATTRIBUTE_DEFINITION(Initializer, Prop, EInvalidateWidgetReason::Paint)`) and invalidate on value change; `TPanelChildren` add/remove invalidates `ChildOrder`; `SetVisibility` invalidates `Visibility`. **What a host must do manually**: only `Invalidate()`/`MarkPrepassAsDirty()` when mutating state the widget reads in `OnPaint`/`ComputeDesiredSize` outside official setters (e.g., your own custom SLeafWidget's draw data — analogous to your `draw_fn` trampoline). Without invalidation infrastructure Slate runs the slow path: prepass+paint of every visible widget every frame. Caching layers: `SInvalidationPanel` (caches children's draw elements; cached children skip prepass/tick/paint), Retainer Panel (renders subtree to texture; high repaint overhead), and **Global Invalidation** (`Slate.EnableGlobalInvalidation 1`, UE 4.25+, replaced LayoutCaching) which treats each `SWindow` as an invalidation root and disables inner invalidation boxes. VALORANT shipped global invalidation on UE 4.25 (+4.26/4.27 backports): up to 15% client FPS, ~35% faster UI rendering — "the single largest performance gain for clients since launch". Frequently-changing widgets should be marked volatile (`ForceVolatile(true)`) so their paint isn't cached. Note 5.7 still has fast-path crash reports (`FSlateInvalidationRoot::ClearAllFastPathData` ensure) — heavy dynamic child churn under global invalidation is the fragile spot.

### 4. Memory/lifecycle
Pure `TSharedRef/TSharedPtr/TWeakPtr` reference counting; no GC, no UObject. Parents own children strongly through slots; the child's back-pointer is `SWidget::ParentWidgetPtr`, a `TWeakPtr<SWidget>` (`AssignParentWidget`/`GetParentWidget`/`ConditionallyDetatchParentWidget`) — so no cycles, and destroying a window cascades destruction. Dangling risks: (a) delegates bound with `CreateRaw`/`CreateLambda` capturing raw `this` outlive the object — bind with `CreateSP` (weak, via `TSharedFromThis`) instead; (b) UMG↔Slate boundary leaks via `FUserWidgetPool` if not released in `ReleaseSlateResources` ("prevent leaking due to circular references"). A reconciler holding its own `TSharedPtr` shadow tree keeps detached widgets alive for free — which is exactly what enables pooling/reuse. Pooling precedent: `SListView/STreeView/STileView` create row widgets only for visible items via `OnGenerateRow` and recycle them as items scroll (100+ item list → a handful of live rows); UMG `UListView` uses `FUserWidgetPool` (`GetOrCreateInstance`/`Release`/`ResetPool`) to pool UUserWidget+SWidget pairs. Reuse-by-type + `AttachWidget` swapping is a legitimate Slate idiom.

### 5. Layout/paint
Two-pass layout: Pass 1 prepass/CacheDesiredSize bottom-up (`ComputeDesiredSize` on leaves, compounds aggregate); Pass 2 `ArrangeChildren` top-down. Paint: `OnPaint(FPaintArgs, FGeometry AllottedGeometry, FSlateRect, FSlateWindowElementList& OutDrawElements, int32 LayerId, ...)` emitting `FSlateDrawElement::MakeBox/MakeText/MakeLines` with integer LayerId z-ordering. Per-frame order: Prepass → Tick → Paint (`FSlateApplication::Tick → TickAndDrawWidgets → SWidget::Paint → OnPaint`). Profiling: `stat slate` ("Num Painted Widgets" = widgets executing OnPaint on the game thread). Scale evidence: the entire Unreal Editor is Slate ("pretty much everything visible in Unreal Editor is a Slate widget"); Epic's UMG optimization guide says complex game screens routinely have "well over 1000 children" loaded with only hundreds visible, and warns off-screen (Hidden-not-Collapsed) widgets still cost layout. Practical budget: thousands of live widgets are fine; thousands *painted per frame without invalidation caching* is where slow-path costs bite (hence VALORANT's numbers).

### 6. Events
Bound at construction via `SLATE_EVENT` (e.g. `.OnClicked(this, &S::Handler)`, `.OnClicked_Lambda`, `_Raw`, `_Static`); handlers return `FReply::Handled()/Unhandled()` (plus `.CaptureMouse()` etc.). **Rebinding after construction works**: delegates are plain assignable members with setters on many widgets — `SButton::SetOnClicked(FOnClicked::CreateLambda(...))` is documented community practice for late/loop binding; `SBorder` exposes `SetOnMouseButtonDown/Up/Move/DoubleClick`; editable text exposes text-change/commit delegate setters. Where no setter exists, a host can standardize on binding once to a per-node trampoline that dispatches to the current diffed handler (same pattern as your Godot signal mapping).

### 7. TAttribute polling vs setters
Legacy `TAttribute<T>` bound to a delegate is **polled every frame** (Slate's stated philosophy: "polling over notification... CPUs are fast and cheap"); Epic's own optimization guide now reverses this for scale: bound attributes "poll the attribute every frame", prefer event-driven updates "on the frame when the value changes". UMG-Slate-Compendium: "TAttributes have a high memory overhead and are not cache friendly." UE 5.0 introduced `TSlateAttribute`/`TSlateManagedAttribute` (member attributes updated once per frame during prepass, registered with an invalidation reason) to make attribute updates fast-path-compatible. **For a diff-based host: pass constant TAttributes (a constant attribute stores a value, no delegate call) and write via setters on change — this is the optimal Slate citizenship**, it keeps widgets non-volatile and cache-friendly under global invalidation, and it's the same model UMG uses.

### 8. Standalone/editor host surface
Standalone Slate apps (SlateViewer at `Engine/Source/Programs/SlateViewer`, UnrealPakViewer, custom launchers): `FSlateApplication::InitializeAsStandaloneApplication(GetStandardStandaloneRenderer())` → `FSlateApplication::Get().AddWindow(SNew(SWindow)...)` → loop `while (!IsEngineExitRequested()) { PumpMessages(); Tick(); FTicker::GetCoreTicker().Tick(dt); sleep-to-60fps; }` → `FSlateApplication::Shutdown()`. In-editor tools use the same widget layer via `FGlobalTabmanager` nomad tabs / `SDockTab`, or raw `SWindow` + `AddWindow`; `FSlateApplication` also offers `FindPathToWidget`, `SetAllUserFocus`, `PushMenu`. Widget Reflector (`Ctrl+Shift+W`, Tools > Debug) inspects any live hierarchy and jumps to the `SNew` source line — the file/line baked into `SNew` exists for this.

**Bottom line for a reconciler host**: creation (`SNew` into shared refs), mutation (setters + slot `AttachWidget`), child diffing (`InsertSlot/RemoveSlot`, keyed reuse via detach/attach), event rebinding (`SetOnClicked`-style or trampolines), and standalone/editor mounting are all first-class. The two engine-boundary hazards are (1) no public slot reorder → remove+insert with `ChildOrder` cost, favor keyed widget reuse over structural moves, and (2) invalidation correctness under `Slate.EnableGlobalInvalidation` when churning children every frame — mark hot leaves volatile or wrap churn regions, and lean on setter-driven writes so the fast path stays valid.

## KEY FACTS
- SNew expands to MakeTDecl<WidgetType>(#WidgetType, __FILE__, __LINE__, RequiredArgs::MakeRequiredArgs(...)) <<= WidgetType::FArguments(); SAssignNew adds .Expose(ExposeAs); both end by calling Construct(const FArguments&).
- EInvalidateWidgetReason bit flags: Layout 1<<0, Paint 1<<1, Volatility 1<<2, ChildOrder 1<<3, RenderTransform 1<<4, Visibility 1<<5, Prepass 1<<7; ChildOrder is the most expensive (rated High by Epic) and implies Layout+Prepass.
- Engine widget setters (SetText, SetVisibility, SetColorAndOpacity, SetContent, panel AddSlot/RemoveSlot) invalidate automatically; a host only calls SWidget::Invalidate()/MarkPrepassAsDirty() for custom state read in OnPaint/ComputeDesiredSize. InvalidatePrepass was deprecated for MarkPrepassAsDirty in UE 5.1.
- SBoxPanel (SVerticalBox/SHorizontalBox) exposes AddSlot(), InsertSlot(Index), RemoveSlot(TSharedRef<SWidget>), ClearChildren(), NumSlots(), GetSlot(i); UE 5.0's slot refactor added FScopedWidgetSlotArguments plus runtime slot setters and FSlotBase::AttachWidget/DetachWidget for in-place child swaps; there is NO public slot-reorder API — reorder = remove+insert.
- SOverlay slots draw in add order unless per-slot ZOrder differs (higher ZOrder on top); SGridPanel::AddSlot(Column,Row,Layer); SCanvas/SConstraintCanvas have AddSlot/ClearChildren with position/size slot attributes.
- Global Invalidation (Slate.EnableGlobalInvalidation, UE 4.25+, replacing LayoutCaching) treats each SWindow as an invalidation root; VALORANT shipped it on UE 4.25 with 4.26/4.27 backports for up to 15% client FPS and ~35% faster UI rendering.
- Ownership: parents own children strongly via slot TSharedRef; the child back-pointer SWidget::ParentWidgetPtr is a TWeakPtr<SWidget> — no cycles; delegates should bind via CreateSP (weak) not CreateRaw to avoid dangling.
- Pooling precedent exists in-engine: SListView/STreeView generate rows only for visible items via OnGenerateRow and recycle them; UMG UListView uses FUserWidgetPool (GetOrCreateInstance/Release/ResetPool), which must be released in ReleaseSlateResources to avoid circular-reference leaks.
- UMG itself is the proof-of-concept for a mutating host: every UWidget caches a TSharedPtr to its SWidget, builds it once in RebuildWidget(), then pushes diffs through Slate setters in SynchronizeProperties.
- Legacy TAttribute delegates are polled every frame and are 'high memory overhead and not cache friendly'; UE 5.0 TSlateAttribute updates once per frame in prepass and registers an invalidation reason (SLATE_ADD_MEMBER_ATTRIBUTE_DEFINITION); Epic's guidance for scale is event-driven setter writes — ideal for a diff-based host passing constant attributes.
- Layout is two-pass (prepass/CacheDesiredSize bottom-up, ArrangeChildren top-down) followed by OnPaint emitting FSlateDrawElement lists with integer LayerId z-order; per-frame order is Prepass → Tick → Paint; 'stat slate' reports Num Painted Widgets.
- Standalone Slate host: FSlateApplication::InitializeAsStandaloneApplication(GetStandardStandaloneRenderer()) → FSlateApplication::Get().AddWindow(SNew(SWindow)) → loop PumpMessages()+Tick() at ~60 FPS → FSlateApplication::Shutdown() (pattern used by SlateViewer and UnrealPakViewer).
- SButton::SetOnClicked(FOnClicked) rebinds click handlers after construction; SBorder exposes SetOnMouseButtonDown/Up/Move/DoubleClick; SLATE_EVENT delegates accept _Lambda/_Raw/_Static/CreateSP bindings and handlers return FReply::Handled()/Unhandled().
- Epic's UMG optimization guide: complex screens have 'well over 1000 children' constructed with only hundreds visible; whole Unreal Editor UI is Slate widgets (inspectable via Widget Reflector, Ctrl+Shift+W); off-screen Hidden (vs Collapsed) widgets still incur layout cost.
- Fast path internals: SWidget::Invalidate routes through FastPathProxyHandle.MarkWidgetDirty() to FSlateInvalidationRoot, which keeps a dirty list and a flattened widget list (BuildFastPathWidgetList); ChildOrder churn rebuilds that list, and UE 5.7 still has crash reports in FSlateInvalidationRoot::ClearAllFastPathData under heavy churn.
