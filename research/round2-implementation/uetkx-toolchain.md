# .uetkx Toolchain Design Digest (grounded in the ReactiveUI-Godot toolchain)

## PART 1 — Ground truth: how the .guitkx toolchain actually works (what we are porting)

### 1.1 Compiler (`addons/reactive_ui/guitkx/`, pure GDScript, engine-light)
- Pipeline: `guitkx_lexer.gd` (`RUIGuitkxLexer`) → `guitkx_markup.gd` / `guitkx_jsx_scan.gd` → `guitkx_codegen.gd` (`RUIGuitkxCodegen`, entry point) → `guitkx_formatter.gd`. Output is a **sibling file**: `Foo.guitkx -> Foo.gd` (`gd_path_for()`), git-ignored, owned by the host language's own compiler — the load-bearing insight ("an EditorImportPlugin can't make preload() a runnable class, but a real sibling source file is one Godot's compiler owns → genuine .new()/render()/hot-reload").
- API shape: `compile_file(guitkx_path, known_components: Array, component_paths: Dictionary) -> {ok, path, gd_path?, diagnostics?, error?, gd_parse_ok, env_error?}`; `compile_all(root) -> {compiled, errors, held, removed, total, bindings}`; `find_all`. `project_bindings(paths)` does ONE read pass producing `{known, bindings (class -> generated path, doubles as HMR link table AND emitter's cross-component path source), losers, sources}`.
- **Lexer is the load-bearing scanner**: everything that balances delimiters (the `return (...)` markup window, `{expr}` attributes, `<Tag>` children) routes through `skip_noncode` so quotes/comments/braces inside embedded code never confuse balancing. It is explicitly "the GDScript-**lexis** port of uitkx's string/char/comment-skipping state machine" — i.e., in this family the markup grammar is constant and the **embedded-language lexis is swapped per host** (C# lexis in Unity, GDScript lexis in Godot, C++ lexis for Unreal). GDScript lexis handled: `#` line comments, `"…"/'…'`/triple-quoted strings, `r"/&"/^"/$"/%"` prefixes (prefix only at token start, not after a value — `a % "f"` is an operator). Perf note [G-10]: per-char reads use `unicode_at` + int constants; the TS mirror keeps `src[i]`; **the shared corpus pins BEHAVIOR, not code**.
- Post-write verification: the generated `.gd` is parsed immediately on a throwaway `GDScript.new()` + `reload()` (class_name line stripped to avoid registry collision) — Unity parity where Roslyn fails fast. Failed compile (T1.1) **deletes the stale sibling** so old output can never silently run.
- Error catalog: `GUITKX####` codes (2101 missing markup return, 2106 duplicate class binding — incumbent keeps compiling, copies error until renamed, 2107 dangling component reference, 2507 environment hold, 0304 unclosed return).

### 1.2 Diagnostics sidecar — the compiler↔LSP contract
- `Foo.guitkx -> Foo.guitkx.diags.json`, **written ALWAYS, even on compile failure**. Schema v2: `{ "v":2, "src_hash", "diagnostics":[{code, severity:int(0 err/1 warn/2 hint), message, off, len}], "refs":{class->generated path} }`. `off`/`len` are character offsets (`off:-1` = whole file); LSP ranges via `positionAt()`.
- `src_hash` = **FNV-1a over code points, constants 2166136261 / 16777619, masked to 32 bits — MUST stay identical to the LSP's `srcHash` (diagsSidecar.ts)** so the LSP can suppress stale sidecars once the buffer diverges. Sidecar not rewritten when payload identical (LSP watches sidecars).
- The sidecar doubles as the **staleness memory**: mtimes are whole seconds, so an mtime TIE is broken by content hash; a file whose last compile ERRORED has no `.gd` at all — its sidecar's (same hash + error entry) means "already compiled and reported broken" → NOT stale (prevents 2s busy-recompile of broken files forever).

### 1.3 Staleness / sweep machinery (all field-capture-hardened; port every rule)
- `is_stale()`: missing `.gd` → stale unless error-verdict sidecar matches; zero mtime (editor scan-window flake) → stale (safe direction); src newer → stale unless standing error verdict for identical content; tie → content hash.
- `has_stale(root)`: cheap read-only poll predicate — dir walk + mtime reads; also counts as work: sourceless sidecars, **orphaned outputs** (generated `.gd` whose source vanished — detected by AUTO-GENERATED header + missing source), and **refs/2107 mismatches** (deleting a component's whole folder leaves no orphan; dependents' sidecar `refs` are checked for missing files both directions: unflagged-but-missing needs the flag, flagged-but-restored needs the heal recompile).
- **Compiler fingerprint**: FNV-1a over the CRLF-normalized concatenation of `_COMPILER_SOURCES` (all pipeline files + vocabulary.json), persisted to machine-local `res://.godot/rui_guitkx_compiler.fp`. Mismatch/absence → every output treated stale (a `git pull` of the compiler otherwise leaves every `.gd` newer-than-source forever; CI hides this by regenerating unconditionally). Fingerprint returns "" (UNKNOWABLE) when any read is empty — never persisted, keeps forcing until readable. Marker refreshed ONLY when nothing was held.
- **Environment hold**: an existing file reading back empty, or vocabulary unreadable (editor first-scan window) → `env_error`/GUITKX2507, outputs KEPT (deleting once wiped every demo `.gd` on a fresh CI clone), one warning per episode, files land in `held[]`, plugin schedules a 2s retry timer (a cold open once stayed uncompiled 3.5h waiting for a user edit).

### 1.4 Watcher / compile-on-save (`addons/reactive_ui/plugin.gd`)
Three triggers because each alone missed field cases:
1. `filesystem_changed` — unreliable: `.guitkx` isn't a Godot resource type, an external-editor-only edit may never flip the flag.
2. `NOTIFICATION_APPLICATION_FOCUS_IN` — recompile on every tab-back into the editor; `is_stale()` mtime guard keeps it cheap.
3. **Standing 2s watch poll** (`_POLL_SECS := 2.0`) over `has_stale()` — covers "save in VS Code while Godot sits in the background" (a stale file once survived 40 min + a restart unnoticed).
Plus: initial sweep **waits out the editor's first filesystem scan** (`is_scanning()`, 0.5s recheck, visible "still waiting" line every ~10s); `_busy` re-entrancy guard with a **30s deadman** (a crashed sweep must cost one 30s outage, not the session); first sweep prints its summary unconditionally ("cold-open proof of life — a silent Output means the plugin is NOT running, never 'nothing to do'").
- **Diagnostic de-dup** (`_last_diags`: path → last-reported body): Godot's Errors dock is append-only, so a persistently-broken file recompiling on every trigger would spam; push each distinct error set once, print a green "compile errors resolved" line when it heals; entries for deleted files forgotten (else a recreated file's first error is suppressed). Format: `path:LINE:COL: CODE: message` (1-based), one dock line per diagnostic.
- Rename hygiene: dock `files_moved` cleans old-name outputs **synchronously** (rapid renames stacked two `.gd` declaring the same class_name and wounded the global class registry).

### 1.5 HMR (Fast Refresh) — `editor/hmr_debugger.gd` (EditorDebuggerPlugin) + `core/hmr.gd` (`RUIHmr`)
- Editor pushes `[gd_paths, bindings]` over debugger channel `"rui_hmr:reload"` after every sweep; game ACKs before work (`rui_hmr:ack` — delivery proof, so a crash in apply can't masquerade as non-delivery), applies **synchronously inside the callback** (no frame boundary between script swap and commit — old-code signal handlers can never fire in a half-swapped world), replies `rui_hmr:status` `{reloaded, reset, refreshed, linked, global, ms, errors}` printed next to the sweep lines — the full save→screen loop visible in one place. "→ 0 session(s)" is always printed (silent logs hid exactly that question).
- Reload is **in place**: `source_code` + `reload(true)` on the SAME script resource so method-reference Callables keep identity and dispatch new code; fiber matching is Callable equality → **hook state survives for free**; `RUIReconciler.hmr_refresh_all` defeats the bailout cache.
- Semantics (Unity RefreshRuntime parity): component changed → targeted re-render (state preserved); hook/shared-module changed → **global** re-render; **hook SIGNATURE changed (`__RUI_HOOK_SIG` const emitted by the compiler since 0.8.0) → deliberate state RESET**, counted and reported; reload failure → per-file isolation (keep old code, report, continue). `bindings` enables hot-LINKING components created after launch (class name unresolvable until next run → resolve by path).

### 1.6 LSP server (`ide-extensions/lsp-server/src/`) — architecture
Files: `server.ts` (bootstrap), `scanner.ts` (TS port of the lexer), `markup.ts`, `jsxScan.ts`, `declScan.ts`, `context.ts` (cursor classification), `schema.ts` + `vocabulary.json` (HOST_TAGS, STRUCTURAL_ATTRS, COMMON_ATTRS, PREAMBLE_DIRECTIVES, CONTROL_FLOW, STYLE_KEYS, BUILTIN_MEMBERS), `virtualDoc.ts`, `sourceMap.ts`, `analyzerAdapter.ts` (in-process napi `@gdscript-analyzer/core`), `classdb.ts`, `events.ts` (onXxx→signal), `workspaceIndex.ts`, `refs.ts`, `declarations.ts`, `liveMarkup.ts`, `semanticTokens.ts`, `diagsSidecar.ts`, `formatGuitkx.ts`, `reflowEmbedded.ts`, `guitkxFormat.ts`, `codePoints.ts`, tests under `src/test/`.
- **Split-brain by design**: markup intelligence answered locally from the schema; embedded-code intelligence answered by an analyzer over a **virtual document + source map** — "no running Godot editor, no TCP, fully offline". Transport: **stdio** (VS Code client and VS2022 `ILanguageClient` both speak it). Full compiler diagnostics arrive via the **sidecar**, gated by `src_hash`.
- **virtualDoc.ts** (the pattern to instantiate a third time): scope-aware — emits the REAL control-flow structure (`for x in xs:`, `if cond:`, `match`) so loop/branch variables are in scope; each `{expr}` becomes `var _eN = (expr)` INSIDE its block, recursively nested; markup/glue never copied; embedded code spliced **VERBATIM (length-preserving)** so offsets map by constant delta. `HOOK_STUBS` (23 entries) emitted as class-level static wrapper funcs whose `params`/`ret` are **byte-identical to hooks.gd, asserted by a parity test** — a bare-Callable alias erased signatures. Misspelled decl keywords are recovered (`findDecl(..., true)`) so a typo doesn't make the whole file go dark.
- **sourceMap.ts**: spans `{sourceStart, genStart, length}`, bidirectional `toGenerated`/`toSource`, offsets not positions ("Volar's technique, hand-rolled — no framework"; analog of Unity's SourceMap.cs).
- **Contract corpus** (`test-fixtures/scanner-cases.json`): sections `skipNoncode`, `findMatching`, `skipNoncodeMarkup`, `findMatchingMarkup`; each case `{input, at, expect}`; **run by BOTH `tests/guitkx_test.gd` and lsp-server `core.test.ts`** — the proven mechanism keeping two implementations in two languages in lockstep. Same pattern for the formatter: `formatter-cases.json` golden corpus pins `formatGuitkx.ts` byte-identical to `guitkx_formatter.gd`.
- The **in-Godot editor plugin** (`addons/reactive_ui_editor/lsp/`) is already a *second* full instantiation of the virtual-doc/source-map/schema stack in GDScript (guitkx_virtual_doc.gd, guitkx_source_map.gd, guitkx_schema.gd, …) with a native analyzer GDExtension bridge and graceful "markup-only stays a fully-supported baseline" degradation. `.uetkx` is thus the **third instantiation of a pattern with two working precedents**.

### 1.7 Formatter + config
- `formatGuitkx.ts` / `guitkx_formatter.gd`: AST-driven re-emit; **returns source VERBATIM on any parse error (never corrupts); idempotent**; `fellBack` flag distinguishes "already canonical" from "couldn't parse". Options `FmtOptions{printWidth:100, indentStyle:"space", indentSize:2, singleAttributePerLine:false, insertSpaceBeforeSelfClose:true}` (Unity-exact defaults). Embedded GDScript blocks remain tab-indented (GDScript requires it); `embeddedReflow` optionally runs embedded code through the analyzer's formatter.
- **`guitkx.config.json` Prettier-style walk-up** (`loadFormatterConfig(fileDir)`): walk up to filesystem root, first hit wins, `"formatter"` section overrides, unknown keys ignored, malformed file → `{}`. Analog of Unity's `uitkx.config.json`.

## PART 2 — .uetkx design decisions

### (a) Compiler language & host — DECISION: Option 3, with the C++ compiler as the single codegen authority
- **C++ compiler lives in an `UncookedOnly` module (`ReactiveUIToolchain`)** — loaded by the editor and by commandlets, never cooked into shipping builds, zero external dependencies. This is non-negotiable for Fab customers: a Fab plugin that requires Node.js to build a game is dead on arrival, and UBT/commandlet integration must be native.
- **TypeScript stays for IDE intelligence only** — exactly the existing division of labor: in Godot, the TS side NEVER generates `.gd`; it scans, parses markup, formats, builds virtual docs, and reads the compiler's sidecars. Reuse `ide-extensions/lsp-server` wholesale: `markup.ts`, `jsxScan.ts`, `context.ts`, `sourceMap.ts`, `semanticTokens.ts`, the schema machinery, and `formatGuitkx.ts` port over nearly unchanged because **the markup grammar is byte-compatible**. The one genuinely new TS piece is a **C++-lexis scanner** (`cppScanner.ts`): `//` and `/* */` comments (no `#` comment — but `#include`/`#if` preprocessor lines need skipping as lines), `"…"` strings with escapes, `'c'` char literals, `R"delim(…)delim"` raw strings, `L/u8/u/U` prefixes; no GDScript `&"/^"` prefixes. This mirrors precisely how `guitkx_lexer.gd` was "the GDScript-lexis port of uitkx's C# state machine".
- **Lockstep mechanism**: a new `uetkx-scanner-cases.json` (+ `uetkx-formatter-cases.json`) contract corpus with the same `{input, at, expect}` shape, **run by both the C++ compiler's automation tests (UE `IMPLEMENT_SIMPLE_AUTOMATION_TEST` or a `-run=RUISelfTest` commandlet) and lsp-server `node --test`**. This pattern already keeps GDScript+TS byte-identical in production; two implementations is a solved problem in this family, three would not be — so the C++ side is the ONLY codegen/diagnostic authority, and the TS side's structural diagnostics stay a "light pass" exactly as today, with the full catalog delivered via `Foo.uetkx.diags.json` (schema v2 verbatim: same fields, same **FNV-1a 2166136261/16777619 src_hash**, same stale-suppression, `UETKX####` code space mirroring `GUITKX####` numbering).
- Rejected: TS-only compiler (Node dependency for every build/CI/Fab consumer; can't run inside the editor watcher without shipping Node); C++-only (loses the existing VS Code/VS2022 server, and marketplace extensions embedding a per-platform native compiler binary is exactly the packaging pain the guitkx `.vsix` already suffers — the uetkx path REMOVES the napi addon need instead, see (d)).

### (b) Build integration — DECISION: committed generated code + stable-TU aggregation + editor watcher + commandlet; PreBuildSteps only as a stale *check*
**Output shape**: `Foo.uetkx -> Foo.uetkx.inl` (sibling, like `Foo.gd`) containing a reflection-free component definition (a function/functor calling the `V::` builder API + a registration static), plus one **stable, always-existing translation unit per module**, `<Module>.Uetkx.gen.cpp`, whose contents are a regenerated `#include` list of every `.inl` in the module. Rationale, in order of force:
1. **UBT makefile-caching pitfall (the flagged critique)**: UBT caches its action graph ("UBT Makefile", `Intermediate/Build/.../*.bin`) including the enumerated source-file set. It invalidates when files are added/removed *between* invocations (directory-enumeration check on load), but a file **created during PreBuildSteps of the same invocation is not in the already-loaded makefile** — the classic two-build problem; deletions leave stale actions. UHT avoids this only because UBT special-cases its generated-code directory (`.gen.cpp` under `Intermediate/Build/<Platform>/<Target>/Inc/<Module>/UHT/`) — **third-party codegen gets no such hook**. The stable-TU design makes the compiled source-file set **constant**: new/removed components only change the *contents* of `<Module>.Uetkx.gen.cpp` and the `.inl` files, which normal timestamp dependency checking handles perfectly. No `-ForceUnity` games needed (the aggregator is effectively a manual unity file; set nothing special in `.Build.cs`).
2. **No UHT ordering problem because generated code is reflection-free**: components are plain C++ (no `UCLASS`/`UPROPERTY` in generated output — the runtime targets Slate directly and interops with UMG via wrappers hand-written in the runtime module). If generated headers carried reflection macros, UHT would have to see them before UBT compiles, recreating the ordering/caching swamp. This constraint goes in the codegen spec.
3. **Generated files are COMMITTED by default** — the deliberate inversion of Godot's git-ignore. Godot can always regenerate headlessly (`godot --headless --script tests/guitkx_build.gd`); UE cannot: the regeneration host is the editor/commandlet binary, which doesn't exist yet on a cold clone of a project being built for the first time, and Launcher-engine Fab customers cannot add engine `Program` targets (the UnrealHeaderTool-style standalone-tool route requires source-engine placement — rejected for Fab). Committed outputs make cold clone, CI without a prebuilt editor, and Fab consumption all work. Staleness safety net: the compiler fingerprint (port of `_COMPILER_SOURCES`/`rui_guitkx_compiler.fp`, keyed on plugin version + toolchain source hash, stored in `Saved/ReactiveUI/compiler.fp`) plus per-file src-hash sidecars; the editor sweep regenerates on open, and CI runs the commandlet and **fails on diff** (drift gate).
4. **Commandlet**: `UnrealEditor-Cmd.exe <Project>.uproject -run=RUICompile [-full] [-check]` — CI/batch entry, `-check` exits non-zero if any output is stale/differs (the CI drift gate); also `-run=RUIExportSchema` (see d). PreBuildSteps in the `.uplugin`/`.uproject` are used ONLY for an optional lightweight stale-warning script (never generation) — a PreBuildStep that invokes `UnrealEditor-Cmd` is circular when the editor target itself is what's being built.
5. **Editor compile-on-save** (port of `plugin.gd`, module `ReactiveUIEditor`): `FDirectoryWatcher` (DirectoryWatcher module) on all module Source dirs containing `.uetkx` + **the 2s `HasStale` poll** (directory watchers miss network drives/WSL mounts — the Godot field captures prove polling is the only trigger that never lies) + window-activation recompile (`FSlateApplication::OnApplicationActivationStateChanged`, the FOCUS_IN analog). Port verbatim: busy guard with 30s deadman, first-sweep proof-of-life summary, env-hold retry, mtime-tie content-hash break, orphan cleanup on rename (via `FDirectoryWatcher` rename events), duplicate-binding incumbent rule, error-verdict sidecars preventing busy loops. Diagnostics → `FMessageLog("ReactiveUI")` with the `_last_diags` de-dup + green "resolved" notification (MessageLog can clear pages, but keep de-dup anyway — the discipline is what makes persistent errors legible).
6. Compiled-C++ refresh during dev: after regenerating `.inl` files, optionally trigger **Live Coding** (`ILiveCodingModule::Get().Compile()`/console `LiveCoding.Compile`) — works because generated code is reflection-free (Live Coding's no-new-UPROPERTY limit never applies). This is the secondary loop; the primary mid-PIE loop is the interpreter (c).

### (c) The dev-loop interpreter — DECISION: interpret structure + restricted expressions; compiled C++ for bodies; state keyed by hook-signature hash; limitations stated plainly
**What happens on save of `Foo.uetkx` mid-PIE** (in-process; no debugger channel needed — unlike Godot, editor and PIE share a process):
1. Watcher sweep → `ReactiveUIToolchain` parses; diagnostics → sidecar + MessageLog (dedup). Parse error → **per-file isolation**: keep the old definition running, report, continue (RUIHmr rule).
2. Parse OK → regenerate `Foo.uetkx.inl` (for the next real build) AND hand the markup AST + expression ASTs to the **interp registry** (`ReactiveUIInterp` module, compiled out of Shipping via `!UE_BUILD_SHIPPING`).
3. Compute the **hook-signature hash** — port of `__RUI_HOOK_SIG` (emitted by guitkx codegen since 0.8.0): FNV-1a over the ordered sequence of hook-call kinds (+arg arity/shape). The compiler bakes the same constant into the generated `.inl`; the interpreter computes it from the AST. Equal → per-fiber hooks array preserved across the swap; different → **deliberate state reset**, counted and reported ("N state reset: hook shape changed" — exact Godot wording).
4. Swap the definition under the **same component key** (file binding / `@class_name`-equivalent), call `FRUIReconciler::HmrRefreshAll()` to defeat the bailout cache. Component change → targeted re-render; a shared-hooks-module `.uetkx` change → global re-render (RUIHmr semantics).

**Interpretable set** (the interpreter emits the SAME `V::` builder calls the generated C++ emits — one VNode input format into one reconciler, so interpreted and compiled components co-exist in one fiber tree):
- Markup structure: all tags, slots, children, keys.
- Literal props: string/FText/FName, int/float/bool, color hex, enum names (via the schema), style objects of literals.
- `@if`/`@for`/`@match` where the condition/iterable is in the expression language.
- `{expr}` — a **restricted, side-effect-free expression VM** over an `FRUIValue` variant: literals; identifiers resolving to props fields, hook-state values (by declaration order), loop vars, context values; operators `+ - * / % == != < <= > >= && || ! ?:`; member access `.` (UStruct fields via reflection, map keys otherwise); index `[]`; calls ONLY into a whitelist registry (`RUI::RegisterInterpFn` — ships with `FText::Format`-style helpers, math, `LexToString`). No assignment, no `new`, no casts, no templates, no arbitrary UE API.
- Hook CALLS in setup: interpreted structurally (kind + order + initial-value expressions through the VM) — `useState(0)`, `useEffect` with a NAMED compiled callback, etc.
- Event handlers: **re-bindable by name** — pointing `onClicked` at a different already-compiled handler works live; the handler *body* is compiled C++.

**Needs a real compile (falls back gracefully)**: new/edited C++ statements in setup blocks beyond hook calls; new/edited event-handler bodies; any expression outside the VM subset; new includes/types. Fallback behavior: the interpreter keeps rendering with the **last-compiled** behavior for those regions, overlays a dev-only on-screen badge on the affected component, and logs "rebuild required for full behavior" (optionally auto-fires Live Coding, a setting). 

**Honest limitation set — state verbatim in the plan**: (1) the mid-PIE loop is *structural* hot reload, not full C++ hot reload — logic changes ride Live Coding or a rebuild; (2) the expression VM is dynamically typed; type errors in `{expr}` surface at the next real compile, not live; (3) interpreted components are slower and exist only in non-Shipping builds; the shipped product runs ONLY compiled codegen output; (4) preserved state means preserved *hooks arrays* — objects captured inside compiled closures are untouched by definition swaps; (5) hook-signature change resets that component's state by design (feature, not bug — same as Godot/Unity). The Godot precedent shows this honesty is survivable because the status line always says exactly what happened (`reloaded/refreshed/reset/linked/global/ms`) — port that reporting verbatim.

### (d) LSP / IDE — DECISION: reuse lsp-server; clangd child process replaces the in-process analyzer; UHT/commandlet-exported schema replaces classdb; VS Code + VS2022 via the existing stdio hosts, Rider via TextMate now + JetBrains LSP API plugin
- **Third instantiation of virtualDoc + SourceMap**, C++ flavor. The virtual document generated per `.uetkx` (extension pinned as `Foo.uetkx.virtual.cpp`, rooted under the module's Source dir so clangd flag inference works):
  1. `#include` lines copied from the file's preamble directive block (the `.uetkx` grammar's preamble carries the component's includes) — spliced verbatim+mapped.
  2. `#include "ReactiveUI/HookStubs.h"` — a real shipped header declaring the hook API with **exact UE types**: `template<typename T> TTuple<T, TFunction<void(const T&)>> UseState(T Initial);`, `void UseEffect(TFunction<TOptional<TFunction<void()>>()> Effect, ...)`, `FRUIContext`, etc. Analog of `HOOK_STUBS` — and port the **parity test**: stub signatures asserted byte-identical to the runtime's real `Hooks` header (core.test.ts precedent: "hook stub signatures match hooks.gd").
  3. Props struct + component opened as a real function; **real control flow emitted** (`for (auto X : Xs) {`, `if (Cond) {`) so loop/branch vars are in scope; each `{expr}` → `auto _eN = (expr);` inside its block; every embedded span spliced **verbatim, length-preserving** — `sourceMap.ts` is reused byte-for-byte unmodified (it's already engine-free offsets).
- **clangd as a child process** (stdio LSP), proxied by lsp-server: forward completion/hover/definition/diagnostic requests for embedded-C++ cursor contexts, translate positions through the SourceMap both ways. This *replaces* `analyzerAdapter.ts`+`@gdscript-analyzer/core` for `.uetkx` — and removes the **platform-specific `.vsix`** problem for the Unreal extension (no napi addon; clangd is fetched via the standard `llvm-vscode`/bundled binary or user-pointed).
- **compile_commands.json dependency**: clangd needs UE flags. UBT generates it: `UnrealBuildTool -mode=GenerateClangDatabase -project=<uproject> <Target> <Platform> Development` (output at repo root; UE5.1+). The server walks up from the `.uetkx` to find `compile_commands.json`; the virtual doc inherits flags from the nearest real `.cpp` in the same module (clangd's transfer heuristic; force it with a generated `.clangd` fragment if needed). **Graceful degradation is the family rule**: no compile_commands.json → markup intelligence fully works, embedded C++ goes dark with a one-line actionable notice (mirror the Godot editor's yellow "native analyzer not loaded — markup features are unaffected" message, including the how-to-fix).
- **Markup-side schema**: Godot's `vocabulary.json` + classdb dump becomes two sources: (1) a **curated, shipped schema** for the plugin's Slate-backed `V::` vocabulary — necessary because **Slate is not reflected** (SWidget attributes aren't UPROPERTYs; no dump can produce them); this is exactly what `vocabulary.json` already is; (2) a **reflection JSON exporter** — `UnrealEditor-Cmd -run=RUIExportSchema` (or a UHT exporter plugin `UHTExporter` script) walking `TObjectRange<UClass>` for UMG `UWidget` interop wrappers, user USTRUCT prop types, UENUM value lists, and dynamic-multicast delegate signatures for `onXxx` event mapping (the `events.ts` analog: React-style `onClicked` → `FOnButtonClickedEvent`). Output `Saved/ReactiveUI/schema.json`, consumed by lsp-server like the classdb dump (`hasDump/classProperties/classSignals` seam already exists); ship pre-generated engine-builtin schemas per engine version so intelligence works before the user ever runs the commandlet.
- Sidecar diagnostics: `Foo.uetkx.diags.json` v2, `srcHash` gate, watched by the server — zero new design.
- **Editor coverage**: VS Code — extend the existing extension (new language id `uetkx`, new TextMate grammar with C++ embedded-scope injection `source.cpp` so themes highlight embedded code even without the server). VS2022 — the existing `ILanguageClient` stdio host reuses the same server binary; VS2022 is also where UE devs already live, making it first-class, and the critique's platform-vsix pain is gone (no native addon). Rider — day one: TextMate grammar (Rider honors TextMate bundles) + sidecar-driven diagnostics are visible via the build; phase 2: a thin Rider plugin hosting the same LSP via the JetBrains LSP API (2023.2+). In-Unreal-editor `.uetkx` tab (the `reactive_ui_editor` analog) is explicitly a LATER phase; the minimum in-editor surface is: asset-browser visibility for `.uetkx`, double-click → external editor, MessageLog diagnostics, and the watcher.

### (e) Formatter + config — DECISION: `uetkx.config.json` walk-up, tab default, dual implementation pinned by golden corpus, clang-format for embedded reflow
- **`uetkx.config.json`**, Prettier-style walk-up from the file's dir to root, `"formatter"` section, unknown keys ignored, malformed → `{}` — identical loader semantics to `loadFormatterConfig` (the family already has `uitkx.config.json` and `guitkx.config.json`; the name is per-dialect, the shape is shared).
- Options identical: `printWidth:100, indentStyle, indentSize, singleAttributePerLine:false, insertSpaceBeforeSelfClose:true`. **Default `indentStyle:"tab"` for .uetkx** — a deliberate per-dialect default divergence: Godot's tabs were forced by GDScript's significant indentation; Unreal's convention (Epic C++ Coding Standard) is tabs, and embedded C++ blocks should look native to UE devs. (Unity dialect keeps spaces/2.) C++ is indentation-insensitive, so unlike `.guitkx` there is **no hard tab requirement** — it's a formatter default, overridable per project.
- Two implementations, same rule as everything else: C++ formatter in `ReactiveUIToolchain` (drives the commandlet + future in-editor view) and the TS port in lsp-server (instant offline Format Document) — **byte-identical via `uetkx-formatter-cases.json`** golden fixtures run on both sides (formatter-cases.json precedent). Both must keep the invariants: return source VERBATIM on any parse error (never corrupt), idempotent, `fellBack` distinguishable from "already canonical".
- Embedded-C++ reflow: analog of `embeddedReflow` — if a `.clang-format` is found by walk-up, run embedded blocks through clang-format (clangd sidecar can serve this); otherwise embedded code is verbatim.
