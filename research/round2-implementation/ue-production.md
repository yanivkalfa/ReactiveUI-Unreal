# ReactiveUI-Unreal — Production Plugin Research Digest (July 2026)

## 1. Plugin Anatomy

### .uplugin descriptor (canonical source: `Engine/Source/Runtime/Projects/Public/PluginDescriptor.h`, `FPluginDescriptor`)
Key fields:
- `"FileVersion": 3` (required, current format version)
- `"Version"` (integer, monotonic) + `"VersionName"` (display string, e.g. `"1.0.0"`)
- `"FriendlyName"`, `"Description"`, `"Category"`, `"CreatedBy"`, `"CreatedByURL"`, `"DocsURL"`, `"MarketplaceURL"` (Fab listing URL — Fab review expects it populated), `"SupportURL"`
- `"EngineVersion"` — e.g. `"5.6.0"`; Fab per-engine-version zips each carry the matching value; the launcher uses it to gate install targets
- `"EnabledByDefault"`, `"CanContainContent"` (must be `true` if the plugin ships `Content/` assets), `"IsBetaVersion"`, `"IsExperimentalVersion"`, `"Installed"` (set on marketplace-installed copies)
- `"Modules": [...]` — per-module `"Name"`, `"Type"`, `"LoadingPhase"`, plus platform/target scoping. **UE5 renamed the scoping keys**: `WhitelistPlatforms`→`"PlatformAllowList"`, `BlacklistPlatforms`→`"PlatformDenyList"`, `WhitelistTargets`→`"TargetAllowList"` (old spellings deprecated but still parsed)
- `"Plugins": [{"Name": ..., "Enabled": true}]` — declared plugin dependencies. **Fab rule (enforced in review): a Fab plugin must compile with no dependencies on other non-engine plugins, including the same seller's** (forum: "Feedback on Plugin Dependency Rule in Fab Marketplace")
- `"PreBuildSteps"` / `"PostBuildSteps"` — real, documented descriptor fields (`FCustomBuildSteps`): per-host-platform (`"Win64": [...]`, `"Linux": [...]`) arrays of shell commands run by UBT before/after a target build, with variable expansion `$(EngineDir)`, `$(ProjectDir)`, `$(TargetName)`, `$(TargetPlatform)`, `$(TargetConfiguration)`, `$(TargetType)`, `$(ProjectFile)`. Precedent: `proletariatgames/unreal.hx` uses PreBuildSteps to run an external compiler before C++ compilation — **this is the natural hook for a `.uetkx` compile step if not done in-editor**.

### Module `Type` (`EHostType::Type`)
`Runtime`, `RuntimeNoCommandlet`, `RuntimeAndProgram`, `CookedOnly`, `UncookedOnly`, `Developer` (deprecated in favor of the next two), `DeveloperTool`, `Editor`, `EditorNoCommandlet`, `EditorAndProgram`, `Program`, `ServerOnly`, `ClientOnly`, `ClientOnlyNoCommandlet`.
- `Runtime` — loads everywhere incl. shipping.
- `Editor` — editor targets only.
- `UncookedOnly` — loads in uncooked editor **and program** targets; the standard choice for modules containing Blueprint-node/compile-time support that must exist for commandlets and UBT-adjacent programs but never in cooked games (e.g. PopcornFX's `PopcornFXEditor` is `UncookedOnly` + `PreDefault`).
- `DeveloperTool` — loaded in any target with `bBuildDeveloperTools`.
- Suggested split for ReactiveUI-Unreal: `ReactiveUI` (Runtime — VDOM, fiber, hooks, Slate host adapter), `ReactiveUIUncooked` (UncookedOnly — `.uetkx` asset/BP support if needed), `ReactiveUIEditor` (Editor — compile-on-save, asset actions, diagnostics), `ReactiveUITests` (Editor or Runtime with `WITH_DEV_AUTOMATION_TESTS`).

### `LoadingPhase` (`ELoadingPhase::Type`)
`EarliestPossible`, `PostConfigInit`, `PostSplashScreen`, `PreEarlyLoadingScreen`, `PreLoadingScreen`, `PreDefault`, `Default`, `PostDefault`, `PostEngineInit`, `None`. (`Default` = during engine init after game modules; `PostConfigInit` needed only for very early systems like RHI hooks; a UI library should stay `Default`; an editor module that registers asset types often uses `Default` or `PostEngineInit`.)

### Build.cs conventions
- Per-module `<Module>.Build.cs` extending `ModuleRules`: `PublicDependencyModuleNames` / `PrivateDependencyModuleNames`; for Slate UI work: `"Core", "CoreUObject", "Engine", "Slate", "SlateCore"`, editor module adds `"UnrealEd", "EditorFramework", "PropertyEditor", "Projects"` etc.
- `PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs;` — the standard for plugins (creates explicit PCH only if `PrivatePCHHeaderFile` is set, otherwise shares engine PCH).
- IWYU conventions (Epic docs "Include What You Use (IWYU) for Unreal Engine Programming"): every header includes what it needs; `.cpp` includes its matching `.h` first; never include monolithic `Engine.h`/`UnrealEd.h`; no explicit PCH includes. IWYU is enabled by default for Engine and Engine plugins, disabled for game plugins — but **marketplace builds are effectively IWYU-strict** because `BuildPlugin -StrictIncludes` compiles with `-DisableUnity -NoPCH -NoSharedPCH`; code relying on unity-build/PCH include leakage fails there (classic failure mode, e.g. getnamo/SocketIOClient issue #419). Write IWYU-clean from day one and make `-StrictIncludes` the CI default.
- `DefaultBuildSettings = BuildSettingsVersion.Latest` (or pinned e.g. `.V5`) in targets; `IWYUSupport = IWYUSupport.Full`.
- **Dependency direction rule:** Runtime modules must never depend on Editor modules (`UnrealEd` etc.) — UBT errors at cook/package time ("module X of type Runtime cannot depend on module Y of type Editor"); editor-only code inside a runtime module goes behind `#if WITH_EDITOR` with the dependency added conditionally in Build.cs via `if (Target.bBuildEditor)`.
- Every distributed class needs its `MODULENAME_API` export macro or BuildPlugin's precompile pass fails for downstream linkage.
- Marketplace/Fab code standards: each source file should carry the seller's copyright line; plugin must compile warning-clean-ish and error-free via the in-editor **Plugins window → Package...** flow before submission (Fab technical requirements say final debugging must be done via Package… because "plugins will be distributed with the binaries built by Epic's compilation toolchain").

### Content vs code plugin
Content-only plugins have no `Source/`, need `"CanContainContent": true`. Code plugins on Fab "must contain at least one code module" and may include any needed content. ReactiveUI-Unreal is a code plugin possibly with a small `Content/` (demo widgets, example `.uetkx` assets).

## 2. Engine Version Support Strategy (state of mid-2026)

- **Release cadence/actuals:** UE 5.4 (Apr 2024), 5.5 (Nov 2024), 5.6 (Jun 2025), **5.7 (Nov 12, 2025)**, **5.8 released Jun 17, 2026** (Preview 1 mid-May 2026, final announced at Unreal Fest/State of Unreal Chicago). **5.8 is stated to be the last major UE5 release; UE6 targets Early Access end of 2027.** So mid-2026 "current three" = **5.6 / 5.7 / 5.8**.
- **Fab requirements:** new products must support the **latest** engine version; published products must maintain support for **one of the three latest** versions. Unlike the old Marketplace, Fab lets you also list arbitrarily older versions ("back to 4.27" confirmed by sellers) and edit/delete per-version uploads. Code plugins require **one uploaded zip per supported engine version** (each zip's `.uplugin` `EngineVersion` set accordingly).
- **Typical seller window:** 3-5 recent versions (Sentry's OSS plugin CI matrixes 4.27→5.5-era, i.e. ~7 versions; commercial sellers commonly do latest 3 + stragglers). Recommended for a new plugin: **launch on 5.6+5.7+5.8**, drop trailing version when a new one ships.
- **Source-compat techniques:**
  - `#include "Misc/EngineVersionComparison.h"` → `UE_VERSION_NEWER_THAN(5, 6, 0)` / `UE_VERSION_OLDER_THAN(...)` (no `_OR_EQUAL` macro; compose with `!UE_VERSION_OLDER_THAN`). Preferred over raw `ENGINE_MAJOR_VERSION`/`ENGINE_MINOR_VERSION`/`ENGINE_PATCH_VERSION` (from `Runtime/Launch/Resources/Version.h`), which broke naive `MINOR >= N` checks when UE5 reset minor to 0.
  - One branch/tag per engine version is the other common pattern ("Develop Low, Upgrade High": author against the **oldest** supported version, auto-upgrade copies for newer — this is the MuddyTerrain/unreal-ci-cd-for-fab philosophy). For heavy Slate API churn a small `Compat.h` shim header with version ifdefs is the norm.
  - Known compat gotcha for the test code itself: automation-flag spellings changed around UE 5.3 (`EAutomationTestFlags` enum-class conversion; `ApplicationContextMask` became the global constant `EAutomationTestFlags_ApplicationContextMask`) — guard with version macros if supporting ≤5.2; irrelevant if floor is 5.6.
- MSVC pinning matters per version (unreal-ci-cd-for-fab pins: UE5.1→MSVC 14.32, 5.2→14.34, 5.3→14.36, 5.4+→14.38+; a real case: UE 5.4.2 rejected MSVC 14.39). VS2022 with "Game development with C++" workload is the baseline; clang-cl optional.

## 3. Packaging

### Command
```
<Engine>\Engine\Build\BatchFiles\RunUAT.bat BuildPlugin ^
  -Plugin="C:\path\ReactiveUI\ReactiveUI.uplugin" ^
  -Package="C:\out\ReactiveUI-5.7" ^
  -TargetPlatforms=Win64+Linux ^
  -Rocket -StrictIncludes
```
(`RunUAT.sh` on Linux/Mac; `-Rocket` = build against an installed/launcher "Rocket" engine — largely implicit in UE5 but harmless; `-TargetPlatforms` joined by `+`; add `-VS2022` style toolchain selectors via BuildConfiguration.xml when needed.)

### What it does/validates
- Compiles every module for host platform (editor) + listed target platforms, Development (+Shipping for precompile), **as an "Installed" plugin** — catching missing `_API` exports, editor-module leakage into runtime, missing includes (with `-StrictIncludes`: `-DisableUnity -NoPCH -NoSharedPCH`).
- Output folder = distributable plugin: `Source/`, `Content/`, `Resources/`, `Binaries/`, `Intermediate/` + anything matched by **`Config/FilterPlugin.ini`** (`[FilterPlugin]` section, paths relative to plugin root, e.g. `/Docs/...`, `/README.md`). Any extra folder beyond Content/Resources/Source that you want distributed (Docs, ThirdParty sources) must be listed there.
- For **Fab submission** strip `Binaries/`, `Intermediate/`, `Saved/` from the zip ("Unreal Engine-generated Binaries and Intermediate files are not required") — Epic's toolchain builds the launcher-distributed binaries; sellers must verify compilation first via editor Plugins window → **Package...**.
- ThirdParty code: externally compiled static libs/DLLs that don't include UE source are acceptable if placed in a ThirdParty folder and included via FilterPlugin.ini; license documentation is required (reviewers check). Shipping standalone helper executables has marketplace precedent (getnamo's NodeJs-Unreal embeds `node.exe` and talks over stdin/stdout; ".exe Extension & Process Management" listing) — but each binary invites review questions; prefer source.
- File size: keep product zip ≤ 15 GB (above that Fab manually reviews necessity).

## 4. Testing

### Frameworks (2025-2026 topography, per Andrew Fray's April 2025 survey + Epic docs)
- **Automation Framework (classic)** — `#if WITH_DEV_AUTOMATION_TESTS` + `IMPLEMENT_SIMPLE_AUTOMATION_TEST(FMyTest, "ReactiveUI.Core.Reconciler", EAutomationTestFlags::EditorContext | EAutomationTestFlags::ProductFilter)` / `IMPLEMENT_COMPLEX_AUTOMATION_TEST` (parametric via `GetTests`). Flags: contexts `EditorContext/ClientContext/ServerContext/CommandletContext` (+`ApplicationContextMask`), filters `SmokeFilter/EngineFilter/ProductFilter/PerfFilter/StressFilter/NegativeFilter`. Most battle-tested; what marketplace plugins ship.
- **Spec tests (BDD)** — `BEGIN_DEFINE_SPEC(FMySpec, "ReactiveUI.Hooks", flags)` … `END_DEFINE_SPEC` + `void FMySpec::Define()` with `Describe/It/BeforeEach/AfterEach` (+ `LatentBeforeEach` etc.). Mature; no BeforeAll/AfterAll; good parametric support (loops in Define). Docs: "Automation Spec" page.
- **CQTest** — Rare/Sea-of-Thieves framework upstreamed into UE (5.3+): `TEST_CLASS`, fluent assertions, before/after each/all, command builders for latent gameplay. Weakness: **no parametric tests**, sparse docs (Lyra has examples). Good for gameplay-ish integration tests.
- **Low-Level Tests (LLT)** — Catch2-based, module-centric, separate `TestTarget` build targets (docs: "Low-Level Tests in Unreal Engine", "Write/Build and Run/Types of Low-Level Tests", UE 5.3→5.8). Runs without full engine boot; positioned for pure-logic testing (ideal in principle for a VDOM/reconciler core), but community adoption is thin and its "unclear positioning in the Unreal ecosystem" is called out; requires its own target compilation which BuildPlugin doesn't produce. Pragmatic call: use classic Automation/Spec as the CI gate, treat LLT as optional.
- **Functional tests** — `AFunctionalTest` actors in maps, driven via Automation framework or **Gauntlet** (`RunUAT RunUnreal`); relevant later for PIE-level demos, not for the library core.

### Headless CLI (the CI loop)
```
UnrealEditor-Cmd.exe C:\proj\Host.uproject ^
  -ExecCmds="Automation RunTests ReactiveUI;Quit" ^
  -unattended -nopause -NoSplash -NullRHI -stdout -FullStdOutLogOutput -log ^
  -ReportExportPath="C:\out\TestReport"
```
- `Automation RunTests <Prefix>` runs all tests under the naming hierarchy; also `RunTest Test1+Test2`, `Automation RunTest Group:MyGroup`, `Automation RunAll`, `Automation RunFilter <SmokeFilter|EngineFilter|ProductFilter|...>`; `;Quit` exits after.
- `-ReportExportPath` emits **index.json + HTML report**; `-ResumeRunTest` resumes crashed runs. Parse index.json (`succeeded`/`failed` counts) or rely on nonzero exit code on failures. Tests require a host `.uproject` that enables the plugin — keep a minimal `HostProject/` in the repo (this is also what BuildPlugin auto-generates internally).
- `-NullRHI` avoids GPU; editor context tests run fine on headless Windows/Linux boxes.

## 5. CI Reality (GitHub Actions, 2026)

- **Engine access:** UE source & official containers are gated behind linking your Epic account to GitHub (join EpicGames org → access `github.com/EpicGames/UnrealEngine` and `ghcr.io/epicgames/unreal-engine`). Official images (since 4.27, docs: "Quick-Start Guide for Using Container Images"): tags `dev-5.6`, `dev-slim-5.6` etc. — **Linux-only development images, ~35-40 GB even for slim**; pulling requires a GitHub PAT of an Epic-linked account; **UE EULA forbids redistributing the engine/images publicly** (so no public caching of engine layers; private registries/runners only; Guganana's MinimalUE — engine trimmed to ~150 MB for compile-only, UE 5.1 — must stay private for the same reason). Not every 5.x tag is guaranteed present (forum reports `dev-5.6` manifest-unknown for a period).
- **No Windows official containers.** Windows CI = self-hosted runner (or paid large runner + launcher/installed-build on an attached disk). "Unreal does not support building Windows builds from Linux"; Linux targets *can* be cross-compiled from Windows with the clang cross-toolchain. Consequence: the pragmatic plugin matrix is **Linux containers (ghcr dev-slim) for cheap per-PR compile+tests, plus one Windows self-hosted runner with N launcher engines installed for BuildPlugin/Win64 packaging**.
- **GitHub-hosted runner limits:** standard runners have ~14-30 GB free disk — an installed engine (~60-120 GB with symbols) doesn't fit; dev-slim (~35-40 GB) only fits after aggressive "free up disk space" steps (Sentry explicitly does this). Sentry (production OSS precedent, `getsentry/sentry-unreal`): matrix 4.27→5.5, runners `ubuntu-latest`/`windows-2019`/arm64, unrealcontainers-style images, headless tests with `-NullRHI -Unattended -NoPause -NoSplash`, packaging via a `pack.ps1` + snapshot-diff validation of package contents.
- **Self-hosted pattern (recommended by multiple sources: mevlyshkin.com, Fall Damage):** Windows VM (AWS/GCP or on-prem), Actions runner service, VS2022 + pinned MSVC toolchains, engines preinstalled per version, `UNREAL_ENGINE_PATH`-style env var; Fall Damage sizing: lots of cores, 1-2 GB RAM/core, ~300 GB persistent disk per engine incl. incremental build state; auto-scale runners up/down for cost.
- **Third-party actions:** ecosystem is thin/unofficial — `UE5-Build-Project` marketplace action (wraps RunUAT), Guganana/UnrealPluginCIWithGithubActions (reusable `CompilePlugin.yml`), MuddyTerrain/unreal-ci-cd-for-fab (local PowerShell pipeline: `run_pipeline.ps1` + `config.json`, `-UseCache`, `-DryRun`, `Tools\validate_config.ps1`; outputs marketplace-clean zips excluding Binaries/Intermediate/Saved; optional rclone upload; **no Fab submission API exists — final upload is manual**). Epic's own answer is **Horde** (ships with engine source since ~5.4/5.5) — overkill for a plugin.
- **Realistic pipeline for ReactiveUI-Unreal:** per-PR: Linux container job per engine version (5.6/5.7/5.8) → build host project with plugin → run `Automation RunTests ReactiveUI` headless → upload index.json; nightly/release: Windows self-hosted → `RunUAT BuildPlugin -StrictIncludes` per version → zip w/ FilterPlugin.ini extras, strip Binaries/Intermediate → artifact. Engine compile of a plugin this size: minutes, not hours; the dominant cost is engine image pull / disk.

## 6. Fab Publishing

- **Onboarding:** fab.com → "Publish" → accept **Fab Distribution Agreement** → publisher profile + unique **Creator Code** → **Trader Verification** (tax/payout via third-party provider; varies by region; EU DSA trader rules apply). Docs: "Publisher Get Started in Fab".
- **Revenue:** **88/12** seller split. Payout ~30 days after month end, **$100 minimum** (rolls over; forced payout after 1 year). Seller-reported real net on a $49.99 sale: ~$40.50-42.80 after fees/taxes/currency. New-seller holds reported (60-day hold, first ~3 months). Fab licenses for tools: **Personal vs Professional tiers** (Professional priced higher, required above revenue threshold), plus seller-set pricing; sales/discount events via Fab's Sales program.
- **Free products:** fully supported ("list products as free or for-sale"); free listings commonly used as funnel; **verified-purchase reviews only**; reviews filterable by engine version.
- **Listing/submission flow:** create listing → upload per-engine-version zips (code plugin: source, no Binaries/Intermediate) → media gallery → **submit for review** before publish. Review timeline is unpublished; real-world anecdotes for code plugins run **days to 3+ weeks** (documented forum case: 18+ days with guideline back-and-forth). Updates to existing listings re-enter review but are typically faster. Plan for iteration lag; don't gate releases on same-day Fab approval.
- **Rules that bite:** new products must support latest UE version; keep at least one of the 3 latest maintained; **no dependencies on other plugins**; per-version zip with correct `EngineVersion`; AI-generated content must be disclosed (undisclosed → delisting); IP/branding conservative (naming a product after another product/trademark gets rejected — relevant: avoid "React" trademark-adjacent naming in the listing title).
- **OSS + Fab dual distribution:** common and allowed — precedents: Cesium for Unreal (Apache-2.0 GitHub + free Fab listing), PlayFab, Sentry, Rive. Fab's Distribution Agreement is non-exclusive; sellers ask/do this routinely (forum thread "GitHub project distribution and (free) Fab Assets"). Free-on-Fab + GitHub-source is the standard OSS plugin play; Fab gives launcher install into `Engine/Plugins/Marketplace/`, GitHub gives project-level `Plugins/` installs and PRs.

## 7. Hot-Reload Landscape

- **Live Coding** (Live++-based; default-on in UE5; Ctrl+Alt+F11 from editor or IDE) — patches binaries while editor runs, works during editing, **PIE**, and attached packaged desktop builds. Handles: function-body edits, minor changes to existing functions, variable-value changes. **Object Reinstancing** (default on) extends it to structural changes by replacing live object instances; caveats: code holding raw pointers to reinstanced objects must handle `ReloadReinstancingCompleteDelegate`/`ReloadCompleteDelegate`; constructor-set defaults in `.cpp` do NOT update existing instances (header-initializer changes do); changing name/type of `UPROPERTY`/`UFUNCTION` reflected members remains the canonical "restart the editor" case; console/mobile targets unsupported. Community guidance is still "close the editor for header/layout changes" — Live Coding is a body-patcher, not a layout migrator.
- **Legacy Hot Reload** (DLL-swap) — effectively deprecated/discouraged since UE5 launch; notorious for corrupting Blueprint references; Live Coding replaced it as default.
- **Implication for ReactiveUI-Unreal (the plan's core dev-loop argument):** an interpreter/VDOM architecture sidesteps both. `.uetkx` → compiled artifact interpreted by the runtime (or → generated C++ only for shipping): editing markup requires **zero C++ recompile** — an editor-module `IDirectoryWatcher` on `.uetkx` files re-compiles markup and triggers a re-render live, in-editor and in-PIE, exactly like the Godot plugin's focus-in recompile. This is strictly better than Live Coding for UI iteration (no layout-change hazard, works for reflected-data-equivalent changes, sub-second). C++ component-logic edits still ride Live Coding within its limits.

## 8. Companion Tool Distribution (.uetkx compiler / LSP)

- **Precedent A — external LSP + editor as data server (Hazelight UnrealEngine-Angelscript):** VS Code extension (`Hazelight.unreal-angelscript` on VS Marketplace; repo `Hazelight/vscode-unreal-angelscript`) contains a TypeScript **language-server + debug adapter**; it **connects to the running Unreal editor** for type data, compile-on-save errors, and debugging. Proven at shipped-game scale (It Takes Two, Split Fiction). Downside vs our Godot approach: online-only (needs a running editor).
- **Precedent B — plugin ships standalone binaries:** getnamo/NodeJs-Unreal embeds `node.exe`, stdio-pipe protocol; ".exe Extension" plugin sells on Fab — shipping helper executables in `Binaries/`/ThirdParty passes review with source/licensing documentation, but adds friction.
- **Precedent C — build-time hooks:** `.uplugin` `PreBuildSteps` running an external compiler (unreal.hx). Works but shells out; platform-conditional commands required.
- **Recommended architecture given this repo's assets:** (a) the canonical `.uetkx`→C++/artifact compiler lives **inside the editor module** (pure C++, mirrors `addons/reactive_ui_editor` compile-on-save + `FDirectoryWatcher`; also expose it as a **commandlet** — `UnrealEditor-Cmd <proj> -run=ReactiveUICompile` — for CI parity with `tests/guitkx_build.gd`); (b) the existing TypeScript LSP from `ide-extensions/` is reused byte-compatibly for markup intelligence (schema-local, offline), with embedded-C++ intelligence either degraded (markup-only), clangd-delegated, or editor-connected Hazelight-style as a later tier. VSIX distribution via VS Marketplace + OpenVSX exactly as the Godot extension does today. No Node runtime ships inside the UE plugin.

## 9. Docs & Samples Conventions

- `.uplugin` `"DocsURL"` surfaces a Documentation link in the editor's Plugins browser — point it at the docs site (a ReactiveUIUnrealDocs sibling of `ReactiveUIGodotDocs~` matches serious-plugin practice; ecosystem uses GitBook/Docusaurus/custom Vite sites — e.g. Voxel Plugin, Cesium, angelscript.hazelight.se).
- Fab listing best practice (seller retrospective): **product page is the conversion surface** — technical docs, video, compatibility matrix listed per engine version; plan 2-3 updates in first 90 days; keep a Discord/newsletter off-platform.
- **Samples:** distributed either as (1) `Content/` inside the plugin (small demos, requires `CanContainContent`), (2) a **separate free Fab listing** containing a complete project (the common pattern — "complete project" product type), or (3) a GitHub sample repo with the plugin as submodule. Fab requires code-plugin submissions to include an updated **example project** per engine version anyway (sellers report "code plugins always require an updated project to be submitted") — the MuddyTerrain pipeline auto-generates per-version example projects from a lowest-version master for exactly this reason.
- In-editor docs: tooltips on every `UPROPERTY`/`UFUNCTION` (meta `ToolTip`), an editor "Getting Started" tab or `Editor Utility Widget`, and demo maps under plugin Content are the norm for tool plugins.
