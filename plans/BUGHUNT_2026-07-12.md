## HIGH severity

### B11 â€” Removing an asset brush prop dangles the widget's `FSlateBrush*` (use-after-free)
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiWidgetAdapters.cpp:174` (and `:107` for `SBorder`)

`Brush`/`BorderImageBrush` are `TSharedPtr<FSlateBrush>` **owned by the committed props**, but the
adapter treats them as plain value props under the "removed plain props don't reset" family rule. The
`RUI_ROW` macro has no reset branch: when a re-render omits the brush, `SetImage`/`SetBorderImage` is
never called, so `SImage`/`SBorder` keeps the raw `const FSlateBrush*` after the props that owned the
`TSharedPtr` are freed.

- **Scenario.** `if (bShowIcon) P.SetBrush(RUI::Umg::MakeAssetBrush(Texture));` â€” render with
  `bShowIcon=true` (SImage caches the raw pointer), then re-render with `bShowIcon=false`. `HasBrush()`
  is now false â†’ row skipped â†’ SImage still points at the old brush. `CommitUpdate` sets
  `Fiber->Props = PendingProps`; one generation later the old props (sole strong owner) are released,
  destroying the `FSlateBrush` while SImage still references it â†’ **next Slate `Paint` dereferences
  freed memory**. The weak entry in `FRuiAssetBrushRoot` also drops, so the texture becomes GC-eligible.
- **Fix.** Treat these as pointer-backed props that MUST reset on removal: in `ApplyDiff`, when
  `O && O->HasBrush() && !N.HasBrush()`, set a safe default (`FStyleDefaults::GetNoBrush()`); or keep
  the last-applied brush `TSharedPtr` alive on the slate node until explicitly replaced/cleared.
- **Why tests miss it.** No test removes the brush after setting it; the `AssetBrush` test only checks
  the tracked-count baseline. Mitigated (not cured) by the documented `UseMemo`/`UseRef` pattern â€”
  which is presented as a *perf* note, not a *safety* requirement.

### B13 â€” `ApplyPropMap` reads the FRuiValue by destination type, not `V.Kind` â†’ zeroed garbage
`Plugins/ReactiveUI/Source/ReactiveUIUMG/Private/RuiUmgElement.cpp:40`

The reflection prop-map switches on the target `FProperty` type and reads the same-named `FRuiValue`
member **without checking `V.Kind`**. `FRuiValue` stores each kind in a *separate* zero-defaulted member,
so a value whose kind â‰  the property category silently writes the default â€” while still counting as
"applied" and pushing the wrong value into the live widget. This contradicts the header's documented
"type-matched to the FRuiValue kind â€¦ mismatches are skipped."

- **Scenario.** `UPROPERTY(float) Opacity` + `WidgetProps={{"Opacity", 1}}`. An integer literal is
  `Kind==Int, FloatValue==0.0`; `FFloatProperty` branch writes `static_cast<float>(V.FloatValue)==0.0f`
  â†’ **Opacity becomes 0 (widget invisible)**. This is the *common* case: the markup VM only marks a
  number `Float` when it carries an `f` suffix (`UetkxExprVm.cpp:369-378`), so bare `1` or even `1.0`
  parse to `Int`. Symmetric failures for Floatâ†’int32, numericâ†’bool (`false`), etc.
- **Fix.** Gate each branch on `V.Kind` (skip + don't increment `Applied` on mismatch), or do explicit
  numeric coercion reading whichever numeric field is populated (`V.Kind==Int ? double(V.IntValue) :
  V.FloatValue`, etc.). Only String/Text/Name branches currently consult `V.Kind`.
- **Why tests miss it.** `ReactiveUIUmgPropMapTest.cpp` only ever passes kind-matched values.

### B8 â€” `SegmentedControl` controlled selection never highlights
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiSegmentedControl.cpp:22, 32`

Both the initial and the controlled-update `SetValue` calls pass `bUpdateChildren=false`. Because the
value attribute is unbound, `SSegmentedControl` bakes each segment's `IsChecked` as a **static snapshot
at `RebuildChildren()`** (all Unchecked), and `SetValue(..., false)` skips
`UpdateCheckboxValuesIfNeeded()`. So the highlight never moves for a controlled value â€” **only a user
click** (which routes through `CommitValue`) ever repaints. The engine passes `false` at construct only
when `Value` is a *bound* attribute; our unbound-plain-int + `false` is the exact broken combination.

- **Fix.** Pass `bUpdateChildren=true` on the controlled path (both `Construct` after `RebuildChildren`,
  and `SetSelectedIndex`).
- **Why tests miss it.** `ReactiveUISegmentedControlTest` asserts `GetSelectedIndex()==GetValue()`,
  never the checkbox visual state â€” the model is right, the render is wrong.

## MEDIUM severity

### B1 â€” `Presence` positional fallback key collides with user integer keys
`Plugins/ReactiveUI/Source/ReactiveUICore/Private/RuiPresence.cpp:30`

`KeyForChild` returns a raw `FRuiKey(Index)` for unkeyed children â€” byte-identical (`EKind::Int`) to a
user's integer key `FRuiKey(0)`. The reconciler deliberately namespaces positional keys as
`FRuiKey(FName("\x01idx%d"))` precisely to avoid this (documented Godot audit-M1 fix); the line-27
comment claims to mirror that namespace but doesn't.

- **Scenario.** `RUI::Presence({ ChildWithNoKey, RUI::FC(B, ..., FRuiKey(0)) })` â†’ both children map to
  `FRuiKey(0)`; the `Emitted` set already contains it, so **`B` is silently dropped and never rendered**;
  in the multi-render path `IncomingIndex` overwrites, mis-attributing which child is exiting.
- **Fix.** Use the same `\x01idx%d` FName sentinel for the unkeyed fallback.
- **Why tests miss it.** No Presence test mixes unkeyed and integer-keyed children.

### B2 â€” Relative navigation with a query/hash doubles the string
`Plugins/ReactiveUI/Source/ReactiveUICore/Private/RuiRouter.cpp:315-317` (and `UseHref`, `:703-704`)

`ResolvePath` segments only on `/`, so `"child?tab=1"` is embedded whole into the resolved pathname,
then `ExtractQueryHash(To)` appends the tail **again**. `ParseLocation` splits at the first `?`, yielding
`Search="?tab=1?tab=1"` (and `ParseSearch` then corrupts the value to `tab="1?tab=1"`). Hashes double
identically to `"#top#top"`.

- **Scenario.** `navigate("child?tab=1")` from `/parent/page`.
- **Fix.** Strip the `?â€¦#â€¦` tail from `To` before `ResolvePath`, append `ExtractQueryHash(To)` once â€”
  in both `Navigate` and `UseHref`.
- **Why tests miss it.** The only internal caller (`UseSearchParams`) always passes an *absolute*
  path (the `To.StartsWith("/")` branch); router tests only exercise relative-*without*-query.

### B4 â€” Inline-only `$token` references are never resolved
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiStyle.cpp:151`

`BuildEffectiveStyle` early-returns the raw inline dict when there are no classes
(`if (Classes.IsEmpty()) return InlineStyle;`), bypassing the entire `$token` resolution loop. So an
inline `{ ColorAndOpacity: "$accent" }` on a class-less widget stays the literal string `"$accent"` and
the adapter receives a `String` where a `Color` was expected. Adding *any* class fixes it â€” proving the
divergence. Contradicts the header contract (no class-presence exception).

- **Fix.** In the empty-classes branch, when `GActiveTheme != NAME_None`, copy `InlineStyle` into a
  fresh dict and run the same resolution loop (never resolve in place on the caller's shared dict).
- **Why tests miss it.** The stylesheet test only exercises tokens *through a class*.

### B5 â€” Global `//` comment stripping corrupts string values containing `//`
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiStyle.cpp:369-379`

`LoadStylesheet` strips `//`-to-EOL over the whole source with no quote awareness, so a value like
`"img://cdn/bg.png"` truncates at `//`. On a single-line block with no trailing newline, `Src.Left(P)`
deletes the rest of the value, the following declarations, **and the closing `}`** â€” the block-walk then
fails to find `}` and drops the whole class silently.

- **Fix.** Strip comments with a char-by-char scan that tracks whether inside a `"â€¦"` string (same for
  `/* */`).
- **Why tests miss it.** Tests only place `//` *after* values as real comments, never inside a string.

### B6 â€” `ComboBox` swallows genuine user picks reported as `ESelectInfo::Direct`
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiComboBox.cpp:106`

The guard assumes `ESelectInfo::Direct` == "our own programmatic `SetSelectedItem`". But UE 5.6
`SComboBox` also emits `Direct` for real user commits: **click-to-close after keyboard navigation**
(`OnButtonClicked` â†’ `Direct`) and **gamepad Accept**. Those user picks hit `HandleSelectionChanged`
with `Direct`, so `OnSelectionChanged` is skipped and the parent is never told â€” while the combo has
*already* updated its own display (a controlled-component desync).

- **Fix.** Don't disambiguate via `ESelectInfo` (Slate overloads `Direct`). Use a reentrancy guard bool
  set around our own `Combo->SetSelectedItem()` calls; ignore the callback only while that guard is set.
- **Why tests miss it.** The test drives selection via `SetSelection(..., OnMouseClick)`, never the
  keyboard-close or gamepad path.

### B7 â€” `DropTarget::OnDrop` never fires `OnDragLeave` â†’ hover styling stuck on
`Plugins/ReactiveUI/Source/ReactiveUISlate/Public/RuiDragDrop.h:167-180`

On a successful drop the target resets its internal `bIsOver=false` but never invokes `OnLeaveCb`. Slate
does **not** deliver `OnDragLeave` to the widget it was dropped on (still under the cursor â€” verified
against UE 5.6 `FSlateApplication::RoutePointerUpEvent`). So a zone that lit up on `OnDragEnter` stays
lit permanently after the drop, because the only reactive un-hover signal never fires.

- **Fix.** In `OnDrop`, fire `OnLeaveCb` (when `bIsOver`) alongside the `bIsOver=false` reset.
- **Why tests miss it.** The `OnDrop` test sets no enter callback; enter/leave are tested independently.

### B9 â€” `ExpandableArea` fires `OnExpansionChanged` for programmatic (controlled) changes
`Plugins/ReactiveUI/Source/ReactiveUISlate/Private/RuiExpandableArea.cpp:41-44`

`SExpandableArea::SetExpanded` unconditionally broadcasts `OnAreaExpansionChanged`, which the wrapper
forwards to the user's `OnExpansionChanged` with no reentrancy guard. So a parent-driven collapse (e.g.
a "Collapse all" button changing `bIsExpanded`) spuriously fires `OnExpansionChanged` as if the user
toggled it â€” the classic controlled-component feedback footgun (analytics/counters/secondary state).
The skip-when-equal guard prevents an infinite loop but not the one spurious fire.

- **Fix.** Set a suppress flag around `Area->SetExpanded()` in the controlled path; `HandleExpansionChanged`
  early-returns while it's set. (Note: inconsistent with sibling `SegmentedControl`, whose `SetValue`
  does *not* re-fire `OnValueChanged` â€” see the "themes" section.)
- **Why tests miss it.** No test drives a controlled expansion change with a bound `OnExpansionChanged`.

### B12 â€” Activatable screen never re-renders on input-method change
`Plugins/ReactiveUI/Source/ReactiveUICommonUI/Private/RuiActivatableScreen.cpp:76`

`RefreshInputMethod()` samples `UCommonInputSubsystem::GetCurrentInputType()` only at `RebuildWidget`
and `NativeOnActivated` â€” and nothing subscribes to `OnInputMethodChangedNative`. A mid-session
mouseâ†’gamepad switch on an already-active screen never re-runs `BuildTree()`, so `State.InputMethod`
stays stale and every `UseInputMethod` consumer keeps rendering the wrong prompts. Directly contradicts
the header's promise ("re-renders on activation **and input-method changes**").

- **Fix.** Bind `UCommonInputSubsystem::GetOnInputMethodChangedNative()` (when a local player exists) to
  a handler that calls `RefreshInputMethod()` + `Rerender()`; unbind in `ReleaseSlateResources`.
- **Why tests miss it.** The CommonUI suite only exercises active/inactive transitions (no live device
  switch â€” which needs PIE input).

### B14 â€” Embedded-C++ intelligence mixes code-point and UTF-16 offset spaces
`ide-extensions/lsp-server/src/virtualDoc.ts:51`

`virtualDoc` stores `scanFile`'s **code-point** region offset as `srcStart`, but the virtual side and
every server consumer (`doc.offsetAt`/`doc.positionAt`) are **UTF-16**. `server.ts` converts at every
*other* scanFileâ†”doc bridge but not on the embedded path. So any `.uetkx` with an astral-plane char
(emoji, CJK-ext, math glyph) before an embedded `setup`/`hook`/`module` body shifts the map by N (=
astral chars before the region), and hover/go-to-definition land on the **wrong token** (or off the
region into the C++ scaffolding).

- **Fix.** Convert at storage: `srcStart: codePointToUtf16(source, srcAt)` in `virtualDoc.emit()` (both
  `virStart` and `length` are already UTF-16). No `server.ts` change needed.
- **Why tests miss it.** Both embedded test files are pure ASCII (code-point == UTF-16). Gated on clangd
  being present.

### B15 â€” `FeaturesPage` Router example will not compile
`ReactiveUIUnrealDocs~/src/pages/Features/FeaturesPage.tsx:9-21`

The headline router snippet (which I wrote this session) doesn't match the shipped API: (1) **`RUI::Route`

- **Fix.** Rewrite against the real signatures: `RUI::Router({ RUI::Routes({ FRuiRoute{ TEXT("/"),

## PLAUSIBLE (likely, needs a runtime repro / a judgment call)

### P1 â€” `FindGlobalViewModel` default class can't resolve a registered subclass
`Plugins/ReactiveUI/Source/ReactiveUIMVVMBridge/Private/RuiMvvmViewModel.cpp:86 vs 100`. Register stamps
`ViewModel->GetClass()` (the concrete subclass); the default `Find` uses `URuiMvvmViewModel::StaticClass()`;

### P2 â€” In-editor preview runs live compiled child-component effects at edit time
`Plugins/ReactiveUI/Source/ReactiveUIEditor/Private/UetkxPreview.cpp:73`. `FUetkxPreview` mounts a real
reconciler root in the editor with no design-time/PIE gate. Any *compiled* child component the previewed
`.uetkx` references is resolved via `RUI::Named(...)` to its real factory, and `CommitRoot` runs its

documents "mounts a component live" as intentional, and ReactiveUICore never itself derefs a null world â€”
the crash requires a user's child effect to reach into `GetWorld()`/a subsystem. Real inconsistency with
the codebase's "no live code in the editor" stance, but severity is contingent. **Fix (recommended).**
Emit a placeholder/no-op for `Component`-kind children during preview (with a collected note), mirroring
`URuiHostWidget`.

  75â†’79/79). Failure requires a stripped custom engine â€” out of scope, with a documented

## Cross-cutting themes (the root causes worth fixing once)

1. **Controlled-widget event/visual semantics were reinvented per widget â€” and each got it differently
   wrong.** ExpandableArea *over*-fires (B9), ComboBox *under*-fires (B6), SegmentedControl doesn't
   *repaint* (B8). There is no shared "controlled Slate widget" helper (a reentrancy guard around
   programmatic sets + a `bUpdateChildren`/repaint contract). **Recommendation:** a small
   `FRuiControlledGuard` utility + a documented rule (programmatic sets never fire the user event; always
   repaint), applied uniformly.
2. **Pointer-backed props ride the value-prop "don't reset on removal" rule â€” unsafe.** Asset brushes
   dangle (B11) and clobber (B10) because `TSharedPtr<FSlateBrush>` props are diffed/reset like ints.
   The prop system needs a "pointer-backed / must-reset-on-removal" category.
3. **Kind-blind `FRuiValue` reads.** B13 (and latent risk anywhere a value is read by expected type
   rather than actual `Kind`). `FRuiValue`'s separate-members layout makes a wrong-kind read silently
   return a default. Any consumer reading `V.FloatValue`/`V.IntValue` without checking `V.Kind` is suspect.
